# 02 · 记忆生命周期

> Corivo 设计文档 v0.10 · [返回索引](./README.md)

---

## 设计原理

Corivo 的记忆管理借鉴人脑四个核心机制：

**遗忘是默认行为。** 大脑只巩固被反复提取或与重要事件关联的记忆。Corivo 的 block 同理——未被使用的 block 自然衰减。

**每次回忆都在改写。** 人每次想起一件事，都在重新构建这段记忆。Corivo 的 block 每次被查询时，Agent 异步审视其准确性，必要时更新。

**记忆自动整合。** 大脑在睡眠时把零散经历重新组织、提炼模式。Corivo 的心跳引擎持续合并重复内容、提炼上层知识。

**记忆有层级。** 工作记忆 → 短期记忆 → 长期记忆，信息在层级间流动。Corivo 的 block 在不同温度层级间流转。

---

## 核心原则：数据永不删除，只降权

Corivo 不做任何真正的删除操作。衰减的 block 只是降低检索权重、移入更冷的存储层级。如同人脑的"遗忘"不是记忆痕迹消失，而是检索路径断了——某个强触发信号可以重新激活它。

---

## 温度梯度

```
热区 (active)      vitality > 60
│  常规查询范围，Agent 每次都扫描
│  整合进程高频运行
│
温区 (cooling)     vitality 20–60
│  精确匹配时可召回，模糊查询不触达
│  整合进程每日运行
│
冷区 (cold)        vitality < 20
│  仅在用户显式要求深度搜索时穿透
│  整合进程每周运行
│
归档区 (archived)  vitality = 0 且长期未回升
   加密冷备，不参与任何检索
   可导出、可恢复，Agent 日常完全忽略
```

假设用户积累 10,000 个 block，日常查询只需处理热区的几百个。响应速度与 block 总量解耦。

---

## 四个自动更新机制

### 一、衰减（Decay）

每个 block 的 vitality 随时间自然递减，衰减速率按 annotation 类型差异化：

- **凭证类**（密码、密钥）：衰减极慢，半年不用仍在热区
- **决策类**：衰减中等，反映决策可能被新决策覆盖
- **临时笔记**：衰减快，几天不用即进入温区

每次被查询命中、被 Agent 引用、被其他 block 的 refs 指向时，vitality 回升。回升幅度按触达方式区分——被用户直接问到 > 被后台整合扫描到。

### 二、重构（Reconsolidation）

Agent 查询并返回 block 时，异步评估该 block 的准确性：内容是否过时？是否有新信息与之矛盾？annotation 是否仍然准确？

如果需要更新，Agent 创建新版本 block，原 block 通过 `consolidated_from` 记录演化链路。用户查询时只看到最新版本，历史版本永远可追溯。

### 三、整合（Consolidation）

心跳引擎后台定期执行：

- **去重**：多个 block 说同一件事 → 合并为一个更完整的 block
- **提炼**：多个相关细节 → 创建一个更抽象的综合 block，refs 关联原始 block
- **补链**：内容高度相关但缺少 refs → 自动补上引用关系
- **降温**：vitality 低于阈值 → 执行 status 降级

整合频率按温度分区：热区高频，温区每日，冷区每周。

### 四、层级流动

```
输入信息
  ↓
Agent 判断是否值得记住
  ↓ 是
新建 block → active, vitality: 100
  ↓ 多次使用和整合
vitality 稳定 → 长期活跃知识
  ↓ 长期不使用
衰减 → cooling → cold → archived
  ↓ 某天深度搜索命中
vitality 回升 → 重新进入 active
```

---

## 审计与可追溯

所有自动操作留完整日志：每次 vitality 变化记录原因，每次整合记录来源 block ID 和合并逻辑，每次 status 降级记录触发条件。

被整合的原始 block 保留为 archived 状态。用户可通过 `corivo log` 查看任何 block 的完整生命历程。

---

## 设计决策

**为什么不用 TTL？** TTL 是一刀切。密码半年才用一次但很关键，临时笔记一天后就无价值。基于 vitality 的衰减允许按内容性质差异化处理。

**为什么重构不覆盖原内容？** 如果 Agent 判断错误，原始信息就丢失了。保留版本链保证安全性，也为"记忆考古"（追溯决策演变）留口子。

**为什么整合异步执行？** 实时整合影响查询响应速度。后台异步执行，用户无感。
