# 07 · 主动推送

> Corivo 设计文档 v0.5 · [返回索引](./README.md)

---

## 定位

推送是 Corivo 核心流程的第三步，也是唯一让用户直接感知 Corivo 存在的环节。采集无感，学习无感，推送有感——这是 Corivo 跟用户"说话"的时刻。

---

## 核心原则

**宁可不推，不可乱推。** 一条错误的推送比十条漏掉的更伤害信任。推送阈值偏高，宁可错过也不打扰。

**推送是建议，不是打断。** 推送融入用户当前的工作流，不弹窗、不独立通知。

**推送必须带 Corivo 品牌标识。** 用户必须知道是 Corivo 在提醒，而不是 AI 工具自己想出来的。

---

## 四种触发时机（v0.5 更新）

### 一、上下文匹配（实时）

用户正在某个 AI 工具里工作，当前上下文与已有 block 高度相关，但用户没有主动查询。

示例：用户在 Claude Code 里讨论部署方案，Corivo 发现有一条 block 记录了上次部署踩过的坑 → 主动提醒。

实现：MCP Server 在 `corivo.query` 响应中附加关联检索结果；IM Bridge 在采集消息时做快速匹配。同一 block 对同一上下文只推送一次，每次对话最多 2 条。

### 二、时间触发（定时）

block 包含时间信息，到了相关时间点主动提醒。

示例："张三说下周三之前给反馈"→ 到了周三还没有新信息 → 推送提醒。"妈妈的生日是 5 月 12 号"→ 提前两天推送。

实现：心跳循环扫描含时间信息的 block，到期时写入 push block。每个时间触发只推送一次。

### 三、洞察触发（整合发现）

心跳整合过程中发现跨 block 的模式或矛盾。

示例：发现用户最近三个项目都选了 Tauri → 推送"要我把这记为你的默认技术偏好吗？"。发现两条 block 有矛盾 → 推送"这两条信息冲突，要更新哪一条？"

实现：整合任务发现洞察时写入 push block（`annotation = push · insight · pending`）。每天最多 3 条。

### 四、**预测触发（v0.5 新增）**

基于决策模式，在检测到决策情境时主动提供预测建议。

示例：用户在讨论后端框架选型，Corivo 检测到这是决策情境，基于历史模式推送预测。

```
[corivo] 预测：基于你的技术选型偏好（安全 0.9、本地优先 0.8），我建议考虑 SQLCipher 或 Deno
置信度：78%（基于 8 个类似决策）
```

实现：Prediction Agent 在上下文匹配时检查是否存在高置信度的预测，存在则推送。每次对话最多 1 条预测。

## 推送通道与品牌呈现

推送通过已有接入点投递，不建独立通知系统。

**Claude Code / Cursor（MCP Server）：**
```
───────────────────────────────────
[corivo] 提醒：这个项目的测试规范是先写 integration test 再写 unit test
[corivo] 关联：上次部署踩过 Docker 网络配置的坑（blk_a3f29x）
[corivo] 预测：基于你的部署经验，建议先检查 Docker 资源限制（置信度 72%）
───────────────────────────────────
```

**飞书 / Telegram / Slack（IM Bridge）：**
```
corivo：张三说的反馈截止日期是本周三，目前还没有收到更新。要帮你跟进吗？
```
```
corivo：你妈妈的生日是后天（5月12号），去年你送的是鲜花。
```

**CLI：**
```
$ corivo query "部署流程"
[结果] ...

[corivo 提醒] 上次部署后遗留了一个 DNS 配置问题还没解决（blk_c7d92k）
[corivo 预测] 你通常先在测试环境验证，建议先检查测试环境配置
```

## Push Block

推送本身也是 block，annotation 格式为 `push · 触发类型 · 状态`：

```
push · context · delivered
push · time · pending
push · insight · delivered
push · prediction · delivered  ← v0.5 新增
```

投递后 vitality 快速衰减，很快进入归档。但记录保留——Corivo 推了什么、什么时候推的、用户是否采纳。

---

## 反馈闭环

用户对推送的反应会反馈到系统中，让推送越来越精准：

**采纳**：用户使用了推送的信息 → 相关 block vitality 大幅回升，推送模型学到这类推送有价值。预测类推送的置信度上调。

**忽略**：用户没有回应 → 中性信号。

**拒绝**：用户说"不相关"或关闭某类推送 → 降低同类推送触发频率。预测类推送的模式权重下调。

---

## 频率控制

| 类型 | 上限 | 理由 |
|------|------|------|
| 上下文匹配 | 每次对话 2 条 | 过多干扰对话流 |
| 时间触发 | 无上限 | 该提醒的不能漏 |
| 洞察触发 | 每天 3 条 | 信息量大，过多造成过载 |
| 预测触发 | 每次对话 1 条 | v0.5 新增，预测需要高置信度 |

总开关：`corivo config set push.enabled false`。可单独关闭某个类型。

---

## 与心跳的关系

```
心跳循环
  ├── 标注 pending block
  ├── 提取决策模式 (v0.5)
  ├── 重构
  ├── 整合 → 发现洞察 → 写入 push block
  ├── 衰减
  └── 检查时间触发 → 到期 → 写入 push block

用户下次交互
  └── 接入点检查 push · pending → 投递并标注 [corivo]
      └── 同时检查预测触发条件 → 生成预测推送 (v0.5)
```

上下文匹配和预测触发不经过心跳，在查询时实时执行（依赖当前对话上下文，不能延迟）。

---

## v0.5 新增：预测推送质量保证

预测推送是 Corivo v0.5 的核心能力，但也是风险最高的——错误的预测会严重伤害信任。

### 置信度阈值

| 置信度 | 行为 |
|-------|------|
| ≥ 80% | 自动推送 |
| 60-79% | 推送，但标注"仅供参考" |
| 40-59% | 不自动推送，用户询问时才显示 |
| < 40% | 不推送，数据不足 |

### 冷启动保护

用户安装后前 7 天，预测推送置信度阈值提高 20%：

- 正常 80% 触发 → 冷启动期需要 ≥ 96%（几乎不可能）
- 目的：积累足够数据再推送，第一次预测必须准

### 用户反馈强化

| 反馈 | 效果 |
|------|------|
| 采纳 | 该类情境的预测阈值降低 5%（更容易推送） |
| 拒绝 | 该类情境的预测阈值提高 10%（更难推送） |
| 连续 3 次采纳 | 该模式进入"高置信模式"，阈值降低至 60% |

---

## 设计决策

**为什么新增预测触发？** 预测是 v0.5 的核心升级。只在用户询问时提供预测，用户体验是被动的。主动推送预测，让用户感受到"数字孪生在帮我思考"。

**为什么预测推送上限是 1 条？** 预测是高风险推送类型，多一条就多一份干扰风险。每次对话最多一条预测，确保是相关性最高的。

**为什么冷启动期提高阈值？** 早期数据少，预测不准。第一次预测错误会让用户永久关闭这个功能。宁可晚推送，也要推送准。

**为什么预测推送需要置信度标注？** 透明度建立信任。告诉用户预测的置信度，用户知道什么时候该相信、什么时候该怀疑。
