# 11 · 工具调用——从提醒者到执行者

> Corivo 设计文档 v1.0 · [返回索引](./README.md)

---

## 定位

没有工具调用能力的 Corivo 只是一个提醒者——"张三的截止日期到了"，然后用户自己去飞书发消息。有了工具调用，Corivo 变成了执行者——"张三的截止日期到了，我已经帮你催了"。

这是 Corivo 从"记忆助手"升级为"工作伙伴"的关键一步。

---

## 核心原则

**渐进信任。** Corivo 不预设自己有权做任何事。从告知执行开始，随着用户信任的建立逐步放开权限。用户始终掌握控制权。

**做了就说。** Corivo 执行的每一个动作，用户都会看到。没有暗箱操作。

**可撤销。** 凡是 Corivo 做的事，用户都能撤销。

---

## 三级权限模型

### Level 0 · 只读（默认）

Corivo 只能查询信息，不能执行任何外部动作。这是安装后的默认状态。

Corivo 的角色：**提醒者**。

```
[corivo] 提醒：张三的反馈截止日期是明天，他还没回复。
```

用户看到提醒，自己决定怎么处理。

### Level 1 · 告知执行

Corivo 可以执行动作，但每次都先告知用户。用户看到 `[corivo] 已执行：...`，不满意可以撤销。

Corivo 的角色：**主动的助手**——先做了再说，但让你看到。

```
[corivo] 已执行：在飞书给张三发了一条提醒，内容："Hi 张三，方案反馈的截止日期是明天，方便的话今天回复一下？"
[corivo] ↩ 撤回这条消息？输入 corivo undo
```

```
[corivo] 已执行：在你的全局偏好中添加了 "默认使用 Tauri 做跨平台开发"
[corivo] ↩ 撤销？输入 corivo undo
```

适用于：更新 Corivo 自身配置、修改注入规则、发送非关键消息、整理文件。

### Level 2 · 请求执行

对于高风险动作，Corivo 先提出建议，等用户确认后才执行。

Corivo 的角色：**谨慎的伙伴**——想好了告诉你，你说好再动手。

```
[corivo] 建议：要不要在 GitHub 上给 project-A 创建一个 issue，标题 "优化数据库连接池配置"？
  ✓ 执行  ✗ 跳过
```

```
[corivo] 建议：下周一是项目评审，要不要帮你从最近的决策记录生成一份摘要发给团队？
  ✓ 执行  ✗ 跳过
```

适用于：对外发送消息、创建/修改代码仓库内容、发送邮件、涉及金钱的操作。

---

## 渐进信任机制

用户不需要手动切换权限级别。Corivo 通过使用模式自动感知信任水平，但每次升级都需要用户明确同意。

### 信任升级路径

```
安装 → Level 0（只读）
  ↓ 使用 1 周 + 用户采纳过 5+ 条推送
Corivo 提议升级
  ↓
"你已经使用 Corivo 一周了，期间我的 12 条提醒中有 8 条对你有帮助。
 要不要让我开始帮你处理一些简单的事情？我会在每次执行后告诉你。
 你随时可以撤销或关掉这个能力。(y/n)"
  ↓ 用户同意
Level 1（告知执行）
  ↓ 使用 1 个月 + 用户未撤销过执行动作
Corivo 提议部分升级到 Level 2
  ↓
"过去一个月我帮你执行了 34 个动作，你没有撤销过。
 有些需要你确认的事情（比如发消息催人），要不要我直接帮你做？
 涉及代码提交和金钱的操作仍然会先问你。(y/n)"
  ↓ 用户同意
Level 2（部分请求执行）→ 高风险动作仍需确认
```

**降级随时可用**：

```bash
corivo config set tools.level 0    # 回到只读
corivo config set tools.level 1    # 回到告知执行
```

## 工具调用的实现

### 通过 MCP 调用外部工具

Corivo 作为 MCP client，连接用户已配置的 MCP server 来执行动作。这复用了整个 MCP 生态——用户已经在 Claude Desktop / Cursor 里配置的 MCP server，Corivo 都可以使用。

```
Corivo 心跳 / 推送触发
  ↓ 决定需要执行一个动作
检查权限级别
  ↓ 允许执行
调用对应的 MCP server
  ↓
  ├── 飞书 MCP → 发消息、创建日程
  ├── Gmail MCP → 发邮件、搜索邮件
  ├── GitHub MCP → 创建 issue、提交 PR
  ├── Slack MCP → 发消息、更新状态
  ├── Google Calendar MCP → 创建日程、查询空闲
  └── 文件系统 → 更新 CLAUDE.md、整理本地文件
```

### 通过 CLI 执行本地操作

不涉及外部服务的动作，直接通过本地命令执行：

```bash
# Corivo 自己调用自己的 CLI
corivo save --content "..." --annotation "..."     # 存储新 block
corivo update blk_xxx --content "..."              # 更新 block
corivo eject --file ~/project/CLAUDE.md            # 管理注入规则
```

### 通过 IM Bridge 执行消息操作

IM Bridge 已经有发消息的能力（原本用于回复用户查询），Corivo 复用这个通道主动发送消息：

```
corivo → IM Bridge → 飞书 API → 给张三发消息
```

## 动作类型与默认权限

| 动作类型 | 示例 | 默认权限 | 理由 |
|---------|------|---------|------|
| 管理自身 | 更新 block、调整偏好、归档过时信息 | Level 0（静默） | 这是 Corivo 管理自己，不影响外部世界 |
| 更新用户文件 | 修改 CLAUDE.md 注入规则、更新全局偏好 | Level 1（告知） | 改了用户的文件，但可撤销 |
| 发送内部消息 | 在飞书/Slack 群里发提醒 | Level 1（告知） | 有外部影响但风险低 |
| 发送对外消息 | 给外部联系人发邮件 | Level 2（请求） | 代表用户对外沟通 |
| 代码操作 | 创建 GitHub issue / 提交 PR | Level 2（请求） | 影响代码仓库，不可轻率 |
| 涉及金钱 | 续费服务、下单 | Level 2（请求） | 财务操作，必须确认 |

## 执行日志与撤销

每一个执行动作记录为 block（`annotation = action · 类型 · 状态`）：

```
action · message · done      ← 给张三发了提醒消息
action · config · done       ← 更新了全局偏好
action · message · undone    ← 用户撤销了发送的消息
```

```bash
# 查看最近的执行记录
corivo actions

最近执行（7 天内）：
  3月18日 10:23  发送飞书消息给张三（催反馈）           ✓ 完成
  3月18日 14:05  更新全局偏好：默认框架 Tauri           ✓ 完成
  3月17日 09:15  发送飞书消息给李四（会议提醒）          ↩ 已撤销

# 撤销最近一次执行
corivo undo

# 撤销指定动作
corivo undo blk_action_xxx
```

## 场景示例

### 日常场景：催人

```
心跳检测到：block "张三承诺周三给反馈" 的时间即将到期
  ↓
权限检查：Level 1（告知执行）
  ↓
Corivo 通过飞书 MCP 发送消息给张三
  ↓
用户在 Claude Code 中看到：
[corivo] 已执行：在飞书给张三发了提醒："Hi 张三，方案反馈的截止日期是明天，方便的话今天回复一下？"
[corivo] ↩ 撤回？输入 corivo undo
```

### 工作场景：准备周报

```
每周五下午，Corivo 发现有一条指令 block："每周五下午生成周报"
  ↓
Corivo 从本周的 block 中提取：
  · 3 个决策
  · 5 个完成的任务（从 GitHub MCP 获取）
  · 2 个待跟进的事项
  ↓
权限检查：Level 2（请求执行，因为要发给团队）
  ↓
[corivo] 建议：你的周报已经草拟好了，要发到飞书团队群吗？

  本周摘要：
  · 确定了 Corivo 的存储方案（SQLCipher + E2EE）
  · 完成了采集模块的设计文档
  · 待跟进：张三的反馈、王五的 API 设计评审
  
  ✓ 发送  ✎ 编辑后发送  ✗ 跳过
```

### 跨工具场景：自动同步偏好

```
Corivo 整合发现：用户最近 3 个项目的 CLAUDE.md 都手动添加了 "使用 pnpm 而非 npm"
  ↓
权限检查：Level 1（告知执行）
  ↓
Corivo 在全局偏好中添加此规则，并同步到所有已注入的 CLAUDE.md
  ↓
[corivo] 已执行：检测到你在 3 个项目中都使用 pnpm，已添加为全局偏好并同步到所有项目配置
[corivo] ↩ 撤销？输入 corivo undo
```

## 设计决策

**为什么默认只读而非直接开放执行？** Corivo 在用户心中还没有建立信任的时候就开始"替你做事"，会让人不安。从提醒开始，让用户先看到 Corivo 的判断是准确的（推送有价值），然后再放开执行权限。信任是挣来的，不是预设的。

**为什么渐进升级需要用户明确同意？** 暗中升级权限是最破坏信任的行为。Corivo 会主动提出"要不要让我开始帮你做事"，但用户说 no 就是 no，不反复纠缠。

**为什么执行后要显示撤销入口？** 告知执行的核心是"做了但你能反悔"。如果撤销不方便，用户会焦虑"它做错了怎么办"。一行 `corivo undo` 消除这个焦虑。

**为什么通过 MCP 调用外部工具？** 复用已有生态。用户在 Claude Desktop 里配的飞书 MCP、Gmail MCP，Corivo 直接拿来用。不需要为每个外部服务写集成代码——MCP 是通用的工具调用协议，Corivo 只需要做好 MCP client。

**为什么高风险动作始终需要确认？** 即使用户信任 Corivo 到了 Level 2，发邮件、提交代码、涉及金钱的操作仍然要确认。这些动作的错误成本太高，不值得用便利性去冒险。一个靠谱的伙伴知道什么时候该自己做主，什么时候该问一声。
