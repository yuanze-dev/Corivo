# 03 · 存储与同步

> Corivo 设计文档 v0.10 · [返回索引](./README.md)

## 核心原则

**数据主权在用户手里。** 所有 block 明文只存在于用户自己的设备上。同步过程中服务器只接触密文。Corivo 团队和任何第三方都无法读取用户数据。项目开源，用户可审计、可自托管。

## 本地存储：SQLCipher

每台设备运行一个加密的 SQLite 实例（SQLCipher），Agent 直接在本地操作 block。

- 查询毫秒级响应，无网络依赖
- 全平台支持：macOS/iOS（原生）、Android（原生）、Web（可选，WASM SQLite）、Windows/Linux
- 语义搜索：sqlite-vss 扩展做本地向量索引，或由 Agent 直接调 LLM 做语义匹配（个人数据量级足够）

## v0.10 核心：密钥管理与设备生命周期

### 密钥架构

```
┌─────────────────────────────────────────────────────────────┐
│                        Corivo 密钥体系                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  主密钥 (Master Key)                                         │
│  ├── 派生方式：PBKDF2(user_password, salt, 100000 iterations)│
│  ├── 存储位置：Keychain (macOS/iOS) / Keystore (Android)    │
│  │               / DPAPI (Windows) / 秘密文件 (Linux)        │
│  └── 用途：加密数据库主密钥                                  │
│                                                             │
│  数据库密钥 (Database Key)                                   │
│  ├── 生成方式：随机生成 256-bit                              │
│  ├── 加密存储：由 Master Key 加密后存储在配置文件中          │
│  └── 用途：SQLCipher 数据库加密                              │
│                                                             │
│  同步密钥 (Sync Key)                                         │
│  ├── 生成方式：从 Master Key 派生                            │
│  ├── 存储位置：中继服务器（加密后的公钥）                    │
│  └── 用途：E2EE 同步的密钥交换                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 设备注册流程

```
新设备首次安装 Corivo
  ↓
用户输入主密码
  ↓
1. 派生 Master Key
2. 生成新的设备公私钥对
3. 如果是第一台设备：
   └── 创建新的同步密钥对
   └── 私钥加密存储在本地
   └── 公钥上传到中继服务器
4. 如果不是第一台设备：
   └── 用户需要从已授权设备获取授权码
   └── 通过授权码从中继服务器获取同步公钥
   └── 验证后完成设备注册
  ↓
设备注册完成
```

### 授权码机制

已授权设备生成授权码：

```bash
$ corivo device authorize

授权码已生成，有效期 10 分钟：

CORIVO-AUTH-XXXXXXXXXXXX

请在新设备上输入此授权码完成注册。
```

授权码格式：
- 一次性使用
- 10 分钟有效期
- 包含：设备签名 + 时间戳 + 随机数
- 中继服务器验证后下发同步公钥

## 设备撤销

### 场景

- 设备丢失或被盗
- 设备退役
- 安全怀疑

### 撤销流程

```bash
# 查看所有已授权设备
$ corivo device list

已授权设备：
  ✓ iMac (2022-03-15 注册，当前设备)
  ✓ MacBook Pro (2023-06-20 注册，最后活跃: 2分钟前)
  ✓ iPhone (2024-01-10 注册，最后活跃: 1小时前)

# 撤销指定设备
$ corivo device revoke "iPhone"

确认要撤销 iPhone 的访问权限吗？
撤销后，该设备将无法同步数据，且本地数据将无法访问。
(y): y

已撤销 iPhone 的访问权限。

中继服务器已更新设备列表。iPhone 上的 Corivo 将在下一次同步时
检测到撤销状态，并锁定本地数据库。
```

### 撤销后的设备状态

被撤销设备的状态机：

```
正常状态
  ↓ 同步时收到撤销通知
撤销通知接收
  ├── 本地数据库立即锁定
  ├── 所有数据转为只读（可导出，不可修改）
  └── 用户界面显示：设备已被撤销
  ↓
用户选择：
  ├── 导出数据 → 生成未加密的备份文件
  ├── 重新注册 → 需要从已授权设备获取新授权码
  └── 卸载 → 删除本地数据
```

### 中继服务器的撤销广播

```
┌──────────────┐
│ 设备 A 发起   │
│ 撤销设备 B    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 中继服务器   │
│              │  1. 验证设备 A 的权限
│              │  2. 更新设备列表（移除 B）
│              │  3. 广播撤销通知给所有在线设备
│              │  4. 标记设备 B 的会话为已撤销
└──────┬───────┘
       │
       ├──────► 设备 C、D：收到通知，更新本地设备列表
       │
       └──────► 设备 B（下次同步时）：
                     1. 收到撤销通知
                     2. 本地数据库锁定
                     3. 通知用户
```

## 密钥丢失恢复

### 场景

- 用户忘记主密码
- Keychain/Keystore 数据损坏

### 恢复机制

**v0.10 策略：用户自主备份恢复**

```bash
# 安装时生成恢复包
$ corivo init

...
正在生成恢复密钥包...

请妥善保管以下恢复密钥（仅显示一次）：

CORIVO-RECOVER-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX

恢复密钥用于：
  - 主密码遗忘时重置密码
  - 设备丢失后恢复数据
  - 密钥存储损坏时重建密钥链

建议：
  - 将恢复密钥抄写在纸上，存放在安全的地方
  - 不要将恢复密钥存储在任何联网设备上
```

### 恢复流程

```bash
$ corivo recover

请输入恢复密钥：CORIVO-RECOVER-XXXX-XXXX-...

恢复密钥验证通过。

请设置新的主密码：**********
确认主密码：***********

密钥链已重建。

正在从同步服务器恢复数据...
已恢复 1,234 个 block
已恢复 45 个决策模式

恢复完成。
```

### 恢复密钥设计

- 格式：16 个单词，用空格分隔（BIP39 风格，便于手抄）
- 生成方式：从 Master Key 派生（使用 KDF）
- 存储：**不存储在任何地方**，用户自己保管
- 验证：输入恢复密钥后，派生出 Master Key，尝试解密本地数据库；成功则验证通过

### 安全考虑

| 威胁 | 防护 |
|------|------|
| 恢复密钥被他人获取 | 密钥足够长（16 词），暴力破解不可行 |
| 用户丢失恢复密钥 | 提供"影子密钥"选项（需另一台已授权设备） |
| 恢复密钥重放攻击 | 恢复后生成新的 Master Key，旧恢复密钥失效 |

## 多设备同步：E2EE 中继

- 用户密钥在设备端派生（PBKDF2/Argon2），不上传服务器
- Block 变更以加密 diff 推送到中继服务器，其他设备拉取后本地解密
- 中继服务极简——本质是一个存密文的消息队列，无业务逻辑
- 设备不需要同时在线，离线修改自动排队，联网后同步

### 同步协议

```
设备 A                                           中继服务
  │                                                 │
  │  1. POST /sync/push                             │
  │     {                                          │

  │       seq: 123,                                │
  │       changes: [                               │
  │         {                                      │

  │           encrypted: "base64(aes_gcm(data))",  │
  │           nonce: "random",                     │
  │           timestamp: 1234567890                │
  │         }                                       │
  │       ]                                         │
  │     }                                          │
  │ ─────────────────────────────────────────────► │
  │                                                 │ 存储加密变更
  │                                                 │ （不解密）
  │                                                 │
  │  2. POST /sync/pull                             │
  │     {                                          │
  │       device_id: "xxx",                        │
  │       last_seq: 100                            │
  │     }                                          │
  │ ─────────────────────────────────────────────► │
  │                                                 │ 查询 seq > 100
  │                                                 │
  │  3. Response                                   │
  │     {                                          │
  │       changes: [...]                           │
```

## 冲突处理

Corivo 是个人产品，同一用户多设备极少同时编辑同一个 block。默认采用 last-write-wins 策略（按 `updated_at` 时间戳仲裁）。检测到冲突时保留双方版本，由 Agent 判断合并或提示用户。

**v0.10 团队版扩展：** 团队版支持多人同时编辑，采用 CRDT（Conflict-free Replicated Data Types）算法自动合并。详见 [12-team.md](./12-team.md)。

## 设计决策

**为什么 SQLCipher 而非 PostgreSQL？** 最初选型为 PG+pgvector（中心化），引入数据安全需求后改为本地 SQLite。结构化查询能力完全满足 block 的增删改查，向量搜索对个人数据量足够。上层 schema 不受影响。

**为什么不用纯 P2P？** 纯 P2P 要求设备同时在线，NAT 穿透不稳定。E2EE 中继在安全性上等价于 P2P（服务器看不到明文），在体验上等价于云同步（随时可用）。

**为什么不用 ElectricSQL / PowerSync？** 这些引擎不内置 E2EE，加密需要额外包装层。Corivo 的同步需求很简单（个人多设备、block 粒度 diff），自建极简中继代码量更小、可控性更强。

**为什么恢复密钥需要用户自己保管？** 如果 Corivo 代管恢复密钥，就引入了中心化风险。用户自己保管恢复密钥是"真正的本地优先"——即便 Corivo 团队也无法访问用户数据。

**为什么设备撤销后不是立即删除数据？** 撤销后设备可能仍离线，无法立即通知。保留只读访问让用户有机会导出数据。如果用户选择"重新注册"，可以无缝恢复（前提是获得新授权码）。
