# 07 · 主动推送

> Corivo 设计文档 v0.10 · [返回索引](./README.md)

---

## 定位

推送是 Corivo 核心流程的第三步，也是唯一让用户直接感知 Corivo 存在的环节。采集无感，学习无感，推送有感——这是 Corivo 跟用户"说话"的时刻。

---

## 核心原则

**宁可不推，不可乱推。** 一条错误的推送比十条漏掉的更伤害信任。推送阈值偏高，宁可错过也不打扰。

**推送是建议，不是打断。** 推送融入用户当前的工作流，不弹窗、不独立通知。

**推送必须带 Corivo 品牌标识。** 用户必须知道是 Corivo 在提醒，而不是 AI 工具自己想出来的。

---

## 四种触发时机（v0.5/v0.10）

### 一、上下文匹配（实时）

用户正在某个 AI 工具里工作，当前上下文与已有 block 高度相关，但用户没有主动查询。

示例：用户在 Claude Code 里讨论部署方案，Corivo 发现有一条 block 记录了上次部署踩过的坑 → 主动提醒。

实现：MCP Server 在 `corivo.query` 响应中附加关联检索结果；IM Bridge 在采集消息时做快速匹配。同一 block 对同一上下文只推送一次，每次对话最多 2 条。

### 二、时间触发（定时）

block 包含时间信息，到了相关时间点主动提醒。

示例："张三说下周三之前给反馈"→ 到了周三还没有新信息 → 推送提醒。"妈妈的生日是 5 月 12 号"→ 提前两天推送。

实现：心跳循环扫描含时间信息的 block，到期时写入 push block。每个时间触发只推送一次。

### 三、洞察触发（整合发现）

心跳整合过程中发现跨 block 的模式或矛盾。

示例：发现用户最近三个项目都选了 Tauri → 推送"要我把这记为你的默认技术偏好吗？"。发现两条 block 有矛盾 → 推送"这两条信息冲突，要更新哪一条？"

实现：整合任务发现洞察时写入 push block（`annotation = push · insight · pending`）。每天最多 3 条。

### 四、预测触发（v0.5 新增）

基于决策模式，在检测到决策情境时主动提供预测建议。

示例：用户在讨论后端框架选型，Corivo 检测到这是决策情境，基于历史模式推送预测。

```
[corivo] 预测：基于你的技术选型偏好（安全 0.9、本地优先 0.8），我建议考虑 SQLCipher 或 Deno
置信度：78%（基于 8 个类似决策）
```

实现：Prediction Agent 在上下文匹配时检查是否存在高置信度的预测，存在则推送。每次对话最多 1 条预测。

## v0.10 新增：旁路触发机制（Bypass Triggers）

**问题**：Aha Moment 依赖数据积累的自然触发，但用户可能：
- 使用频率低，迟迟达不到触发阈值
- 希望主动验证 Corivo 是否理解自己
- 需要向他人展示 Corivo 的能力

**解决方案**：为每个 Aha Moment 提供人工触发方式。

### 旁路触发接口

```bash
# 触发 Aha #1：用户画像
$ corivo insight profile

✦ Corivo 对你的理解：

  你是一个全栈开发者，偏好 TypeScript strict mode。
  你有 8 个活跃项目，最常用 Claude Code 和 Cursor。
  你的代码风格：ESLint + Prettier，Tab 缩进，单引号。
  你倾向使用 Tauri 做跨平台、PostgreSQL 做后端存储。

  数据来源：12 个项目配置文件，47 条提取记忆
  更新时间：2 分钟前

# 触发 Aha #2：最近记录
$ corivo insight recent

✦ Corivo 最近帮你记录的：

  [今天 14:23] 数据库选型确定用 SQLCipher（决策 · project · corivo）
  [今天 11:05] 与张三约定周三反馈（事实 · people · 张三）
  [昨天 16:42] 周报格式偏好：先决策再进展（指令 · self · 周报）

# 触发 Aha #3：跨对话召回
$ corivo insight connections --query "数据库选型"

✦ Corivo 跨对话关联：

  当前讨论：新项目的数据库选型
  │
  ├── 3 天前：Corivo 项目选择了 SQLCipher（vitality: 95）
  │   理由：local-first + E2EE 需求
  │
  └── 2 周前：my-app 项目选择了 PostgreSQL（vitality: 60）
      理由：团队熟悉，有现成运维经验

  推荐：如果是个人工具，参考 Corivo 的选择；如果是团队项目，参考 my-app

# 触发 Aha #4：主动推送测试
$ corivo insight test-push

✦ Corivo 主动推送模拟：

  你有一条即将到期的承诺：
  "张三说本周三之前给反馈"（2 天后）

  要让 Corivo 在到期前主动提醒你吗？
  [配置提醒 → corivo remind on blk_xxx]

# 触发 Aha #5：跨工具串联
$ corivo insight cross-tool --days 7

✦ Corivo 跨工具串联（最近 7 天）：

  飞书 → Cursor：
    王五在飞书提到 API 限流策略建议用令牌桶算法
    └─ 你现在在 Cursor 里讨论 rate limiting
    └─ [查看上下文]

  Claude Code → 飞书：
    你在 Claude Code 决定了使用 Tauri
    └─ 团队在飞书讨论技术栈时可以参考
    └─ [分享给团队]

# 触发 Aha #6：预测能力
$ corivo insight predict --context "后端框架，需要处理用户敏感数据"

✦ Corivo 预测：

  基于你的决策模式：

  技术选型偏好：
  - 安全性权重：0.9（8 个相关决策）
  - 本地优先权重：0.8（6 个相关决策）
  - TypeScript 偏好：87.5%（7/8 次选择）

  建议：Deno + SQLite
  理由：内置 TypeScript、本地友好、安全可控

  置信度：78%（基于 8 个类似历史决策）
```

### 旁路触发的价值

1. **加速信任建立**：用户不需要被动等待，可以主动验证 Corivo 的能力
2. **冷启动友好**：新用户安装后可以立即看到 Corivo 的价值，而不是等 7 天
3. **演示工具**：用户可以用 `corivo insight` 向他人展示 Corivo 的能力
4. **调试工具**：开发者可以用来验证不同功能是否正常工作

### 旁路触发限制

| 功能 | 自然触发 | 旁路触发 | 说明 |
|------|---------|---------|------|
| 用户画像 | 安装后自动 | 随时可用 | 数据不足时显示"仍在学习中" |
| 最近记录 | 第一次自动记录后 | 随时可用 | 无记录时显示"还没有记录" |
| 跨对话召回 | 第 2-3 天 | 随时可用 | 数据不足时显示"积累不够" |
| 主动推送 | 第 3-7 天 | 随时可用 | 模拟推送，不发送真实通知 |
| 跨工具串联 | 第 1-2 周 | 随时可用 | 需要接入两个数据源 |
| 预测 | 第 2-4 周 | 随时可用 | 数据不足时显示"需要更多决策数据" |

## 推送通道与品牌呈现

推送通过已有接入点投递，不建独立通知系统。

**Claude Code / Cursor（MCP Server）：**
```
───────────────────────────────────
[corivo] 提醒：这个项目的测试规范是先写 integration test 再写 unit test
[corivo] 关联：上次部署踩过 Docker 网络配置的坑（blk_a3f29x）
[corivo] 预测：基于你的部署经验，建议先检查 Docker 资源限制（置信度 72%）
───────────────────────────────────
```

**飞书 / Telegram / Slack（IM Bridge）：**
```
corivo：张三说的反馈截止日期是本周三，目前还没有收到更新。要帮你跟进吗？
```
```
corivo：你妈妈的生日是后天（5月12号），去年你送的是鲜花。
```

**CLI：**
```
$ corivo query "部署流程"
[结果] ...

[corivo 提醒] 上次部署后遗留了一个 DNS 配置问题还没解决（blk_c7d92k）
[corivo 预测] 你通常先在测试环境验证，建议先检查测试环境配置
```

## Push Block

推送本身也是 block，annotation 格式为 `push · 触发类型 · 状态`：

```
push · context · delivered
push · time · pending
push · insight · delivered
push · prediction · delivered
push · bypass · manual    ← v0.10 新增：旁路触发
```

投递后 vitality 快速衰减，很快进入归档。但记录保留——Corivo 推了什么、什么时候推的、用户是否采纳。

---

## 反馈闭环

用户对推送的反应会反馈到系统中，让推送越来越精准：

**采纳**：用户使用了推送的信息 → 相关 block vitality 大幅回升，推送模型学到这类推送有价值。预测类推送的置信度上调。

**忽略**：用户没有回应 → 中性信号。

**拒绝**：用户说"不相关"或关闭某类推送 → 降低同类推送触发频率。预测类推送的模式权重下调。

**v0.10 新增：旁路触发反馈**

```bash
# 旁路触发后收集反馈
$ corivo insight predict --context "后端框架"

[预测结果...]

这个预测有用吗？
[1] 非常有用  [2] 有点用  [3] 没用  [4] 错误

# 或自动记录反馈
$ corivo feedback blk_a3f29x --rating useful
```

## 频率控制

| 类型 | 上限 | 理由 |
|------|------|------|
| 上下文匹配 | 每次对话 2 条 | 过多干扰对话流 |
| 时间触发 | 无上限 | 该提醒的不能漏 |
| 洞察触发 | 每天 3 条 | 信息量大，过多造成过载 |
| 预测触发 | 每次对话 1 条 | 预测需要高置信度 |
| **旁路触发** | **无限制** | **v0.10：用户主动调用，不限** |

总开关：`corivo config set push.enabled false`。可单独关闭某个类型。

---

## 与心跳的关系

```
心跳循环
  ├── 标注 pending block
  ├── 提取决策模式
  ├── 重构
  ├── 整合 → 发现洞察 → 写入 push block
  ├── 衰减
  └── 检查时间触发 → 到期 → 写入 push block

用户下次交互
  └── 接入点检查 push · pending → 投递并标注 [corivo]
      └── 同时检查预测触发条件 → 生成预测推送

用户主动调用（v0.10）
  └── corivo insight * → 旁路触发 → 立即显示结果
```

上下文匹配和预测触发不经过心跳，在查询时实时执行（依赖当前对话上下文，不能延迟）。

旁路触发是用户主动调用，不经过心跳，直接执行。

---

## v0.5/v0.10 预测推送质量保证

预测推送是 Corivo v0.5/v0.10 的核心能力，但也是风险最高的——错误的预测会严重伤害信任。

### 置信度阈值

| 置信度 | 自然触发 | 旁路触发 |
|-------|---------|---------|
| ≥ 80% | 自动推送 | 直接显示 |
| 60-79% | 推送，标注"仅供参考" | 显示，标注"仅供参考" |
| 40-59% | 不自动推送 | 显示，标注"数据不足" |
| < 40% | 不推送 | 显示，标注"数据严重不足" |

### 冷启动保护

用户安装后前 7 天，预测推送置信度阈值提高 20%：

- 正常 80% 触发 → 冷启动期需要 ≥ 96%（几乎不可能）
- 目的：积累足够数据再推送，第一次预测必须准

**旁路触发不受冷启动限制** —— 用户主动查看预测时，即使数据不足也显示，只是标注"数据不足"。

---

## 设计决策

**为什么新增预测触发？** 预测是 v0.5/v0.10 的核心升级。只在用户询问时提供预测，用户体验是被动的。主动推送预测，让用户感受到"数字孪生在帮我思考"。

**为什么预测推送上限是 1 条？** 预测是高风险推送类型，多一条就多一份干扰风险。每次对话最多一条预测，确保是相关性最高的。

**为什么冷启动期提高阈值？** 早期数据少，预测不准。第一次预测错误会让用户永久关闭这个功能。宁可晚推送，也要推送准。

**为什么预测推送需要置信度标注？** 透明度建立信任。告诉用户预测的置信度，用户知道什么时候该相信、什么时候该怀疑。

**为什么 v0.10 新增旁路触发？** Aha Moment 的自然触发依赖数据积累，有些用户可能使用频率低，迟迟体验不到核心价值。旁路触发让用户可以主动体验 Corivo 的能力，加速信任建立，也方便向他人展示。
