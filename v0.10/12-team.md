# 12 · 团队与企业版

> Corivo 设计文档 v0.10 · [返回索引](./README.md)
> E1 扩展：团队协作记忆层 · P1 优先级

---

## 定位

个人版 Corivo 是"数字孪生"——记住一个人的决策、偏好、历史。

团队版 Corivo 是"团队记忆"——记住团队的决策、知识、约定。它不是另一个知识库工具，而是团队 Agent 共享的记忆层。

---

## 核心理念

**团队知识应该自然积累，不是手动整理。**

团队每天都在做决策、写代码、讨论问题。这些信息散落在：
- Slack/飞书消息流
- GitHub issue 和 PR
- 文档 Wiki
- 会议记录

Corivo 团队版自动从这些渠道采集信息，提取关键决策和知识，让 Agent 在帮团队工作时能调用团队的全部历史上下文。

**个人是基础，团队是延伸。**

团队版建立在个人版之上，共享 block 机制让个人知识可以转化为团队知识。

## 架构扩展

### 命名空间（Namespace）

个人版只有一个命名空间：`default`

团队版引入多命名空间：

```
┌─────────────────────────────────────────────────────────────┐
│                      Corivo 命名空间架构                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户级别                                                   │
│  ├── default（个人记忆）                                    │
│  └── shared（从团队共享的）                                 │
│                                                             │
│  团队级别                                                   │
│  ├── team:{team-id}（团队记忆）                             │
│  │   ├── 决策类 block：团队技术选型、架构决策                │
│  │   ├── 知识类 block：团队最佳实践、运维手册                │
│  │   └── 指令类 block：团队代码规范、工作流程                │
│  │                                                          │
│  └── team:{team-id}:public（对外公开）                      │
│      └── 可选择性对外共享的部分知识                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Block Schema 扩展

| 新增字段 | 类型 | 默认值 | 说明 |
|---------|------|--------|------|
| `namespace` | string | `"default"` | block 所属命名空间 |
| `shared_to` | string[] | `[]` | 共享给哪些团队/用户 |
| `visibility` | enum | `"private"` | `private`/`team`/`public` |
| `author` | string | `null` | 创建者 ID（团队版必需） |

### 查询策略

```bash
# 个人查询（默认）
$ corivo query "数据库选型"
# 只查询 default 和 shared 命名空间

# 团队查询
$ corivo query --namespace team:corivo "数据库选型"
# 查询团队命名空间

# 跨命名空间查询
$ corivo query --all "数据库选型"
# 查询所有可访问的命名空间
```

## 个人知识 → 团队知识

### 共享机制

用户可以选择将个人 block 共享给团队：

```bash
# 查看个人知识中可共享的内容
$ corivo shareable --team "corivo-dev"

可共享给团队的决策类 block：
  [1] SQLCipher 选型决策（vitality: 95）
  [2] Tauri 跨平台方案（vitality: 88）
  [3] MCP 接入架构（vitality: 76）

# 共享指定 block
$ corivo share --team "corivo-dev" blk_a3f29x

已将 "SQLCipher 选型决策" 共享到 Corivo 开发组

# 批量共享符合条件的内容
$ corivo share --team "corivo-dev" --auto --condition "vitality > 80 AND annotation LIKE '决策%'"

已共享 12 个 block 到团队
```

### 共享后的 block 行为

```
个人 block（原始）
  │
  └── 共享到团队
       │
       ├── 团队成员可查询
       ├── 团队成员可引用（refs）
       ├── 团队成员不能修改原始内容
       └── 原作者可以随时撤销共享
```

### 逆向同步：团队知识 → 个人

```bash
# 发现有价值的团队知识
$ corivo discover --team "corivo-dev"

团队知识推荐：
  [1] 部署流程（作者：alice，23 人引用）
  [2] 代码审查规范（作者：bob，18 人引用）
  [3] Onboarding 清单（作者：carol，15 人引用）

# 添加到个人知识库
$ corivo adopt --team "corivo-dev" blk_team_xxx

已将 "部署流程" 添加到你的个人知识库
```

## 冲突解决：CRDT

个人版采用 last-write-wins，团队版需要更智能的冲突解决：

### CRDT 策略

```
┌─────────────────────────────────────────────────────────────┐
│                      CRDT 合并策略                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  文本内容：RGA (Replicated Growable Array)                  │
│  ├── 追加操作自动合并                                       │
│  └── 冲突编辑保留双方版本                                   │
│                                                             │
│  refs 引用：OR-Set                                          │
│  ├── 添加引用自动合并                                       │
│  └── 删除引用自动同步                                       │
│                                                             │
│  vitality：LWW (Last-Write-Wins)                            │
│  └── 取最大值                                               │
│                                                             │
│  status：状态机                                            │
│  └── 状态转换遵循规则，不允许非法转换                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 冲突提示

```bash
# 检测到冲突
$ corivo sync

检测到 2 个冲突：

[1] block "数据库选型" 有冲突版本
  ├── 设备 A：PostgreSQL（xiaolin，3月18日 14:23）
  └── 设备 B：MySQL（alice，3月18日 14:25）

[2] block "部署流程" 有冲突版本
  ├── 设备 A：三阶段部署（xiaolin，3月18日 15:10）
  └── 设备 B：四阶段部署（bob，3月18日 15:12）

要解决冲突，运行：corivo resolve
```

## 设计决策

**为什么团队版建立在个人版之上？** 每个人先有自己的 Corivo，然后共享有价值的内容给团队。这样团队知识库是自然积累的，不是手动整理的。

**为什么共享是单向的？** 个人 block 共享到团队后，团队成员可以引用但不能修改。修改权在原作者，这保证了知识的源头清晰。

**为什么团队版需要 CRDT？** 多人协作必然产生冲突，last-write-wins 不够。CRDT 让冲突可以自动合并，减少人工介入。

**为什么企业版不强制 E2EE？** 企业有合规需求，可能需要监管访问。E2EE 作为可选配置，企业自己决定安全策略。

**为什么命名空间设计如此简单？** 复杂的权限系统难以理解和使用。Corivo 的命名空间只有三级：个人、团队、公共，覆盖 99% 的使用场景。
