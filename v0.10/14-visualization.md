# 14 · 可视化图谱

> Corivo 设计文档 v0.10 · [返回索引](./README.md)
> E3 扩展：记忆可视化 GUI · P1 优先级

---

## 定位

Corivo 是 CLI 优先的产品，但可视化是理解"数字孪生"的关键。

当用户能**看到**自己的记忆图谱、决策时间线、领域热力图时，Corivo 不再是一个黑盒，而是一个可见的"数字自我"。

---

## 核心理念

**让用户看到 Corivo 在想什么。**

可视化不是为了好看，而是为了：
- 验证 Corivo 是否正确理解
- 发现记忆中的盲点和冲突
- 感受"数字孪生"的存在

---

## 设计原则

**Web GUI，不是本地 GUI**

- 通过浏览器访问 `http://localhost:3000`
- CLI 启动本地服务器
- 无需额外安装

**渐进增强，不破坏 CLI 体验**

- CLI 是核心，GUI 是补充
- 所有 GUI 操作都可以通过 CLI 完成
- GUI 提供 CLI 无法提供的视觉体验

## 实时更新

### WebSocket 推送

```javascript
// 客户端连接
const ws = new WebSocket('ws://localhost:3000/ws')

// 服务器推送更新
ws.onmessage = (event) => {
  const update = JSON.parse(event.data)

  switch (update.type) {
    case 'block.created':
      // 新 block 创建，更新图谱
      graph.addNode(update.data)
      break

    case 'block.updated':
      // block 更新，刷新节点
      graph.updateNode(update.data)
      break

    case 'vitality.changed':
      // vitality 变化，更新热力图
      heatmap.refresh()
      break
  }
}
```

### 事件类型

| 事件 | 说明 | GUI 响应 |
|------|------|---------|
| `block.created` | 新 block 创建 | 图谱添加节点 |
| `block.updated` | block 更新 | 图谱更新节点 |
| `block.deleted` | block 删除 | 图谱移除节点 |
| `vitality.changed` | vitality 变化 | 热力图刷新 |
| `refs.changed` | 引用关系变化 | 图谱连线变化 |

## 设计决策

**为什么 Web GUI 而非原生 GUI？** Web 技术栈开发速度快，跨平台容易。本地 GUI 需要针对每个平台单独开发。

**为什么 D3.js 而非其他图形库？** D3.js 生态成熟，文档丰富，社区活跃。Cytoscape.js 专门用于复杂网络可视化。

**为什么需要 WebSocket？** 心跳引擎会持续更新数据，WebSocket 让 GUI 可以实时反映变化，不需要用户手动刷新。

**为什么提供导出功能？** 可视化是用于理解的，导出是用于分享的用户可能需要把记忆图谱分享给团队成员，或导出报告用于存档。

**为什么 GUI 是可选的？** Corivo 核心 CLI 足够完成所有操作。GUI 是增强体验，不是必需品。保持 CLI 优先可以减少依赖，降低使用门槛。
