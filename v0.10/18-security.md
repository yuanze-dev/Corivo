## 安全原则

**数据主权在用户，安全性是 Corivo 的生存基础。**

任何安全漏洞都可能破坏用户信任，导致项目失败。

## 1. 加密与密钥管理

### 1.1 密钥派生

| 检查项 | 标准 | 验证方法 |
|-------|------|---------|
| PBKDF2 迭代次数 | ≥ 100,000 | 代码审查 |
| 盐值长度 | ≥ 16 字节 | 代码审查 |
| 盐值唯一性 | 每用户不同 | 测试验证 |
| 主密码强度要求 | 最短 8 字符 | 功能测试 |
| 弱密码拒绝 | 拒绝常见密码 | 测试验证 |

**代码检查：**
```typescript
// ✓ 正确
const key = pbkdf2(password, salt, {
  iterations: 100000,
  keyLength: 256,
  hash: 'sha256'
})

// ✗ 错误：迭代次数太少
const key = pbkdf2(password, salt, {
  iterations: 1000,  // 太少
  keyLength: 256
})
```

## 11. 安全开发生命周期

### 11.1 威胁建模

```
┌─────────────────────────────────────────────────────────────┐
│                      威胁建模流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 资产识别                                               │
│     ├── 用户数据（block）                                   │
│     ├── 密钥（Master Key, Sync Key）                        │
│     ├── 访问令牌（API Token）                               │
│     └── 系统配置                                           │
│                                                             │
│  2. 威胁识别                                               │
│     ├── 数据泄露（服务器被黑、本地被盗）                     │
│     ├── 中间人攻击（网络拦截）                               │
│     ├── 权限提升（Token 泄露）                              │
│     ├── 拒绝服务（资源耗尽）                                │
│     └── 侧信道攻击（时序分析）                              │
│                                                             │
│  3. 对策设计                                               │
│     ├── E2EE 加密 → 防止服务器泄露                          │
│     ├── TLS → 防止网络拦截                                 │
│     ├── Token Scope → 限制权限提升                          │
│     ├── 速率限制 → 防止 DOS                                 │
│     └── 常量时间 → 防止时序攻击                            │
│                                                             │
│  4. 验证测试                                               │
│     ├── 渗透测试（每年）                                    │
│     ├── 代码审计（发布前）                                  │
│     └── Bug Bounty（持续）                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 12. 安全事件响应

### 12.1 事件分类

| 级别 | 描述 | 响应时间 |
|------|------|---------|
| P0 严重 | 密钥泄露、数据大规模泄露 | 4 小时 |
| P1 高危 | 身份绕过、权限提升 | 24 小时 |
| P2 中危 | DoS、信息泄露 | 72 小时 |
| P3 低危 | 配置错误、文档问题 | 7 天 |

### 12.2 响应流程

```
检测 → 确认 → 遏制 → 根除 → 复盘
  ↓       ↓       ↓       ↓       ↓
监控    评估影响  紧急修复  修复根本  总结教训
报告    分类定级  通知用户  漏洞扫描  更新流程
```

### 12.3 披露策略

- **P0/P1**：7 天内披露补丁
- **P2/P3**：30 天内披露补丁
- **0-day**：协作者提前 7 天通知

## 设计决策

**为什么安全单独成文档？** 安全是 Corivo 的核心价值主张。任何安全问题都可能导致项目失败，需要特别关注。

**为什么需要威胁建模？** 无法防护不知道的威胁。威胁建模系统化地识别潜在风险，确保防护措施覆盖所有关键攻击向量。

**为什么需要事件响应计划？** 所有系统都有漏洞。关键是如何快速响应和修复。事前准备比事后慌乱更有效。

**为什么需要第三方审计？** 自己的眼睛看不到自己的盲区。独立安全审计是发现深层问题的必要手段。
