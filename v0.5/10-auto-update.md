# 10 · 自动更新

> Corivo 设计文档 v1.0 · [返回索引](./README.md)

---

## 定位

Corivo 作为用户工作流中的常驻伙伴，必须始终保持最新——新的采集器支持、更好的标注算法、更精准的推送模型。用户不应该操心"我的 Corivo 是不是该更新了"。

---

## 核心原则

**用户无感。** 更新在后台发生，不弹窗、不打断、不需要手动操作。

**安全第一。** 更新只从官方源拉取，校验签名后才应用。不自动重启正在运行的服务，等下次自然启动时生效。

**可控。** 用户可以关闭自动更新、锁定版本、选择更新频率。开源用户可以从源码自行构建。

---

## 更新机制

### 搭便车：心跳循环中检查

不需要独立进程。在心跳循环的最低优先级位置，每 24 小时检查一次新版本。

```
心跳循环
  ├── 1. 标注 pending block（最高）
  ├── 2. 重构
  ├── 3. 热区整合
  ├── 4. vitality 衰减
  ├── 5. 温区/冷区整合
  └── 6. 检查更新（最低，每 24h 一次）
```

### 更新流程

```
检查 npm registry 最新版本
  ↓ 有新版本
后台下载到临时目录
  ↓ 校验 checksum / 签名通过
标记为"待应用"
  ↓ 下次 CLI 启动 / MCP Server 重启时
应用更新
  ↓
静默通知用户一次
```

**不中断当前运行。** MCP Server 正在服务 Agent 请求时不打断。等到下一次自然的进程重启时生效——用户关了电脑第二天打开，新版本就在了。

### 更新通知

```
[corivo] 已更新到 v1.2.0 — 新增 Slack 采集器、改进推送精准度
```

不弹窗、不阻塞、只出现一次。

---

## 更新内容分层

| 层级 | 内容 | 更新方式 |
|------|------|---------|
| CLI 核心 | 命令、schema、心跳逻辑 | npm 包更新，重启生效 |
| LLM prompt | 标注/整合/推送的 prompt 模板 | 随 CLI 包更新 |
| MCP Server / IM Bridge | tool 定义、采集器、bot 逻辑 | 随 CLI 包更新，重启生效 |
| 注入规则模板 | CLAUDE.md 等注入的文本 | 更新后提示用户确认同步 |

### 注入规则的特殊处理

更新后注入规则可能也有优化（新增查询语法、改进触发条件）。Corivo 对比已注入内容与新版本模板，有差异时提示：

```
[corivo] 有新的记忆规则可用。要更新你的 CLAUDE.md 吗？(y/n)
```

用户确认后替换注入段落，保留用户自己写的内容。拒绝则不再提醒。

## 用户控制

```bash
corivo version                          # 查看当前版本和更新状态
corivo update                           # 手动触发更新
corivo config set update.auto false     # 关闭自动更新
corivo config set update.pin v1.1.3     # 锁定版本
corivo config set update.interval 7d    # 更新检查频率
```

---

## 设计决策

**为什么搭心跳便车？** 不引入新进程，与整体架构一致。心跳已经在跑，加一个低优先级任务零成本。

**为什么不自动重启？** 正在运行的 MCP Server 可能正在处理请求，强制重启打断用户工作流。等自然重启最安全。

**为什么注入规则更新要用户确认？** CLAUDE.md 是用户的文件。安装时授权了一次，不代表以后每次都有权改。逐次确认保持信任。
