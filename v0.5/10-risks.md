# 10 · 风险与防御

> Corivo 设计文档 v0.5 · [返回索引](./README.md)

---

## 定位

本文档记录 Corivo 面临的主要风险，以及对应的防御策略。风险不是用来吓退自己的，而是用来提前想好对策的。

---

## 战略风险

### 风险 #1：OpenAI/Anthropic 推出官方长期记忆

**可能性**：高
**时间窗口**：6-18 个月

**影响**：ChatGPT/Claude 的用户可能不再需要 Corivo 的记忆功能

**防御**：

1. **跨工具定位是护城河**
   - OpenAI 的记忆只能用于 ChatGPT
   - Corivo 的记忆跨 ChatGPT、Claude、Cursor、飞书
   - 营销语：**"你的数字自我不属于任何单一工具"**

2. **加快开源社区建设**
   - 官方记忆必然是闭源的
   - Corivo 保持开源，用户数据完全自主
   - 吸引重视隐私的用户和开发者

3. **差异化能力**
   - 官方记忆：单一工具内的连续性
   - Corivo：跨工具的数字孪生 + 预测能力
   - 官方很难做跨工具（商业利益冲突）

**应对时间表**：
- 6 个月内：完成核心功能，建立用户基础
- 12 个月内：形成明显的跨工具价值感知
- 18 个月内：建立开源社区和品牌认知

---

### 风险 #2：用户不理解 Corivo 在做什么

**可能性**：高
**后果**：卸载、负面评价

**防御**：

1. **Aha Moment 设计**（详见 09-aha-moment.md）
   - 安装后 60 秒："它已经认识我了"
   - 第一次对话："它在帮我记"
   - 第一次跨会话召回："它记得我说过的话"

2. **透明度**
   - 所有推送带 `[corivo]` 标识
   - `corivo log --today` 让用户看到今天记了什么
   - 每次存储在对话中标注 `[corivo] 已记录：...`

3. **可控性**
   - `corivo sources` 查看所有采集源
   - `corivo sources off xxx` 单独关闭
   - `corivo eject` 一键退出

## 技术风险

### 风险 #3：规则注入被工具更新清空

**可能性**：中
**后果**：Corivo 停止工作，用户不知道

**防御**：

1. **注入标记**
   ```
   <!-- CORIVO BEGIN -->
   ... 注入内容 ...
   <!-- CORIVO END -->
   ```

2. **完整性检测**
   ```bash
   corivo doctor    # 检测所有注入点的完整性
   ```

3. **自动恢复**
   - 通过 Git hook 监听文件变化
   - 检测到注入被破坏时，自动询问是否恢复
   - 提供 `corivo fix` 命令手动恢复

4. **守护进程兜底**（v0.5）
   - 如果所有寄生点失效，启动守护进程
   - 确保 Corivo 始终有运行的心跳

---

### 风险 #4：Pending Block 积压

**可能性**：中
**后果**：查询结果不完整，用户体验卡顿

**场景**：用户有几天不用，pending block 积累到上千条

**防御**：

1. **动态加速**
   - 检测 pending block 数量超过 500
   - 心跳循环增加处理批次（10 → 20）

2. **批量 LLM 调用**
   - 一次处理 10-20 条 pending block
   - 降低 LLM 调用次数

3. **进度可见**
   ```
   $ corivo status
   Pending blocks: 1273 (预计处理时间：15 分钟)
   ```
   - 用户知道进度，不会觉得卡死

4. **降级策略**
   - pending 积压时，优先处理最近的
   - 旧 pending 降低优先级

---

### 风险 #5：性能问题导致用户卸载

**可能性**：低
**后果**：用户流失

**防御**：

1. **性能指标监控**
   - 查询响应时间 < 500ms（p95）
   - 心跳循环执行时间 < 5 秒
   - 数据库大小 < 1GB（正常使用 6 个月）

2. **定期归档**
   - archived block 自动移出主表
   - 保留全文索引，但移出热数据

3. **可选的本地模型**
   - 推荐使用本地 LLM（Ollama）
   - 避免网络延迟影响体验

## 隐私与安全风险

### 风险 #6：隐私悖论

**问题**：Corivo 的定位是"隐私优先"，但：
- E2EE 中继同步需要信任中继服务
- 飞书/Slack 消息采集——这些平台已经有你的数据
- AI 工具的注入规则——AI 提供商可以看到你让 Corivo 记住了什么

**真相**：Corivo 不让数据变少，只是让用户知道数据在哪里

**防御**：

1. **诚实告知**
   - 在隐私文案中明确说明
   - 不夸大"零泄露"能力

2. **本地优先**
   - 核心数据只存在本地
   - 同步是可选的，用户可以不同步

3. **敏感信息处理**
   - 密码、token 等提供明确删除选项
   - 支持设置"敏感类别"，默认不上传

---

### 风险 #7：数据永不删除原则的隐患

**问题**：
- 数据库持续增长 → 查询变慢
- 敏感信息（密码、token）永远保留 → 安全风险
- GDPR "被遗忘权"无法满足

**防御**（v0.5 更新）：

1. **区分"归档"和"删除"**
   - 噪音类 block 可以 30 天后物理删除
   - 敏感信息类别提供明确删除选项

2. **新增 `corivo forget` 命令**
   ```bash
   corivo forget "AWS"              # 删除所有包含 AWS 的 block
   corivo forget --category asset   # 删除所有资产类信息
   corivo forget --older-than 90d   # 删除 90 天前的噪音
   ```

3. **敏感信息检测**
   - 注入时检测密码、token、密钥等模式
   - 标记为 `sensitive`，询问用户是否要加密存储

## 产品风险

### 风险 #8：冷启动的鸡与蛋

**问题**：Aha #4（主动推送）需要 30+ 条完成态 block，但用户第一天只有 0 条

**循环**：需要数据才能推送 → 需要推送才能留存 → 需要留存才有数据

**防御**：

1. **种子数据**
   - 导入 ChatGPT/Claude 历史作为启动燃料
   - 从本地配置文件（CLAUDE.md 等）提取偏好

2. **宽松采集**
   - 第一周宁可多存，之后收紧衰减
   - 降低首次推送的阈值

3. **主动训练**
   - 引导用户手动训练几条偏好
   - `corivo init` 完成后，弹出训练引导

4. **提前预告**
   ```
   $ corivo init
   ...
   [提示] 主动推送将在积累 30 条记录后启用
   通常需要 3-7 天，你也可以主动训练加速
   ```

---

### 风险 #9：预测推送第一次就错了

**可能性**：低（因为有置信度阈值）
**后果**：严重——用户永久关闭预测功能

**防御**（v0.5）：

1. **置信度阈值**
   - 正常情况：≥ 80% 才推送
   - 冷启动期（前 7 天）：≥ 96%

2. **第一次必须对**
   - 第一次预测推送前，人工审核
   - 或者只推送置信度 ≥ 90% 的

3. **快速降级**
   - 一次拒绝：该类情境阈值 +10%
   - 连续 3 次拒绝：该类情境停止预测推送

## 竞争风险

### 风险 #10：竞品抄袭

**可能性**：高（代码开源，设计文档公开）

**防御**：

1. **先发优势**
   - 抢占用户心智：Corivo = 跨工具数字自我
   - 开源社区建设：contributor = 用户

2. **网络效应**
   - 用户越多，模式库越丰富
   - 个人数据迁移成本高（不是竞品能解决的）

3. **持续创新**
   - v0.5：预测性 AI
   - v0.6：代理能力（代替用户执行决策）
   - 保持 1-2 个版本领先

---

## 总结

| 风险类别 | 最高风险 | 防御重点 |
|---------|---------|---------|
| 战略 | OpenAI/Anthropic 官方记忆 | 跨工具定位 + 开源社区 |
| 技术 | 规则注入被清空 | `corivo doctor` + 自动恢复 |
| 隐私 | 隐私悖论 | 诚实告知 + 本地优先 |
| 产品 | 冷启动 | 种子数据 + 宽松采集 |
| 竞争 | 竞品抄袭 | 先发优势 + 网络效应 |

**风险管理的本质**：不是消灭所有风险，而是知道风险在哪里，提前准备好应对方案。
